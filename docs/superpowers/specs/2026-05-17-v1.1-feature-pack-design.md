# Yappr v1.1 — Four-Feature Pack

**Status:** Draft — awaiting user review 2026-05-17
**Authors:** Noan + Claude
**Implementation strategy:** One design doc, four independent specs → four independent implementation plans → four independent launches. Order is suggested, not required.

---

## Background

Competitive research (May 2026) against Wispr Flow and the broader dictation-app category surfaced four asymmetric opportunities — high user demand, clean fit with Yappr's BYOK + creative-polish thesis, and tractable inside the current Electron + Groq + local-Whisper architecture. This document specifies all four; each section is intentionally self-contained so it can be lifted into its own implementation plan without cross-references.

The four features, in suggested launch order:

1. **Lifetime tier** — pricing/positioning. Counters Wispr's $144/yr-forever resentment. Lowest engineering effort.
2. **First-word capture + Bluetooth mic robustness** — fixes two of Wispr's most-cited reliability complaints. Audio-pipeline change.
3. **Editor-aware variable biasing (quick-win)** — large accuracy delta for developers, the audience most hostile to Wispr. Filesystem + tree-sitter.
4. **Context memory** — three stacked layers (focused text-box, global user overview, per-category activity summary). Largest scope; gated behind a hard performance budget.

## Feature 1 — Lifetime Tier

### What it is

A one-time-payment SKU sold alongside the existing free tier and (future) managed monthly tier.

**Pricing:**
- **Lifetime:** $99 one-time, BYOK forever, every feature unlocked.
- **Monthly managed:** price decided when the managed-inference backend is built (placeholder SKU only at launch) — same features *plus* Yappr-managed Groq inference (no key setup), priority support, cloud sync of dictionaries and macros across machines.
- **Free:** unchanged — fully functional with the user's own free Groq key, no time limit.

### Why monthly remains attractive even with a lifetime option

The lifetime tier is a strict BYOK product. Monthly subscribers get three things lifetime users do not:

1. **Managed inference** — no Groq account setup, no key rotation, no per-user free-tier quota anxiety. Just install and dictate.
2. **Cloud sync** — dictionaries, macros, and (eventually) context memory summaries sync across machines. Lifetime stays local-only.
3. **Priority support** — direct response channel; lifetime users get community support.

This keeps monthly attractive to two real cohorts: (a) users who don't want to manage API keys, and (b) multi-machine users who care about sync. Engineering-cohort early adopters take the lifetime tier; mainstream users on their second laptop take monthly.

### Why this matters

The single most-cited Wispr complaint on Trustpilot and Reddit (2.7/5 average) is the absence of a one-time option versus $144/yr-forever. Superwhisper sells lifetime at $249.99. At $99 we undercut both and still pocket essentially all of it — there is no per-user inference cost on the lifetime SKU.

### Scope

This feature is **not engineering work in the desktop app**. It is:

1. Stripe (or Lemon Squeezy / Paddle) product setup: one lifetime SKU at $99 one-time, one monthly SKU as placeholder.
2. License-key issuance flow: user buys → email with key → in-app "Enter license" field in Settings → About.
3. License validation: signed-token check on first activation, offline-friendly afterward (lifetime should never phone home repeatedly).
4. Landing-page copy updates: pricing table with three columns (Free / Lifetime / Monthly), comparison emphasis on managed-inference + sync for monthly.

### Out of scope

- Building the monthly managed-inference backend (separate future project — placeholder SKU only).
- Refund handling automation.
- Enterprise/team plans.

### Risks

- **Cannibalization:** if monthly's value isn't clear, every monthly customer downgrades to lifetime. Mitigation: launch monthly *after* lifetime, with managed inference as the visible difference. Don't list monthly until that backend exists.
- **License-key piracy:** trivial to crack with one-time validation; accept this — lifetime + BYOK means the marginal cost of a pirated copy is zero to us anyway.

### Done means

- Stripe SKU live.
- "Enter license" field in About tab validates and unlocks Pro features. Initially nothing is gated — v1 stays fully free. Whether Features 2/3/4 in this pack get gated to paid tiers is a per-feature decision at launch time, not committed here.
- Landing page pricing table updated.

## Feature 2 — First-Word Capture + Bluetooth Mic Robustness

### What it is

Eliminates two specific Wispr failure modes:

- **Lost first word:** user starts speaking before the hotkey-press → mic stream is fully open, losing the first 100–400ms.
- **Bluetooth mic freeze / cold-start latency:** BT mics have ~200–500ms wake-up latency on macOS; users dictate into a dead stream.

### Mechanism

**Rolling pre-roll buffer.**

A continuously-running 500ms ring buffer of mic audio, held in RAM. When the user presses the hotkey, transcription consumes (last 500ms from ring buffer) + (everything from now until release). The ring buffer overwrites itself in place — audio older than 500ms is gone, never written to disk, never sent anywhere.

Why 500ms not 300ms: cheap (≈16KB at 16kHz mono int16), gives headroom for slow-finger-lift on the hotkey, and absorbs Bluetooth wake-up latency in the same primitive.

**Bluetooth-aware mic path.**

At AudioContext init:

1. Read `MediaStreamTrack.getSettings()` and the device label from `enumerateDevices()`.
2. If device class indicates Bluetooth (AirPods, generic BT headsets — label heuristics + `kAudioDeviceTransportTypeBluetooth` via native helper on macOS), apply two adjustments:
   - Pre-flight 200ms warm-up: open the stream and discard the first 200ms before the ring buffer starts retaining audio (BT mics often emit silence or pops during wake-up).
   - Force 16kHz sample rate — many BT mics report 48kHz but actually capture at 16kHz; the resampling causes glitches on slow CPUs.

### Battery and performance

The user's specific concern: don't drain battery by recording 24/7.

**Reality:** the mic stream is already always open while the indicator pill is mounted (this is how the existing partial-transcript streaming works). The ring buffer adds ~16KB of memory and zero extra wall-clock work — it's the same audio frames the AudioWorklet already processes, just retained in a circular `Float32Array` instead of dropped. Measured impact: indistinguishable from the current implementation.

If we ever wanted to reduce idle drain further, the orthogonal change is to *suspend* the mic when the app is not in focus and reopen on focus — that is **not part of this spec** and would need its own latency budget.

### Implementation sketch

- New module `src/main/audio/pre-roll-buffer.ts` — circular `Float32Array`, 500ms × 16kHz = 8000 samples.
- AudioWorklet `process()` writes every frame into the ring; on hotkey-press, snapshot the ring + concat with subsequent frames.
- New module `src/main/audio/bt-detection.ts` — native helper (Objective-C bridge via `node-mac-permissions` or similar) reading `kAudioDevicePropertyTransportType`. Cache result per device UID.
- Hot path unchanged for non-BT mics: only adds one `Array.prototype.set` per audio frame, sub-microsecond.

### Done means

- Hotkey-press → transcription includes 500ms of audio prior to press timestamp.
- Bluetooth mic users (Noan to test with AirPods Pro) no longer report missing first words.
- No measurable increase in idle CPU or battery drain vs. current main branch (verified via macOS `top -pid` and battery telemetry over 1hr).

### Out of scope

- Adaptive pre-roll length based on user behavior.
- Wired-mic-specific tuning (current behavior is already correct for wired).
- Whisper streaming improvements (the existing local-streaming feature is unchanged).

## Feature 3 — Editor-Aware Variable Biasing (Quick-Win)

### What it is

When the focused app is a known code editor and the focused window title reveals a file path, Yappr extracts identifiers from that file and passes them to Whisper as `prompt` bias. Result: `userId`, `handleSubmit`, `MyComponent.tsx` come out correctly on first dictation, with no user dictionary entry needed.

### Scope (quick-win version)

This is the *quick-win*, not the LSP-integrated version mentioned in the future-features memo. The quick-win path:

1. Detect focused editor (Cursor, VS Code, Windsurf, Zed) — already done in the existing app-detection layer.
2. Parse window title for file path. macOS editors commonly format titles as `filename.ext — project — Editor`. A small regex per editor extracts the filename.
3. If the file is readable from disk (likely — user opened it in their editor, lives in their workspace), parse with tree-sitter for the file's language. Cap at top-150 identifiers by frequency.
4. Pass the identifier list (joined, ~200 token cap) as Whisper's `prompt` parameter for cloud Groq, and as `initial_prompt` for local whisper.cpp.
5. Cache parsed identifiers per `(file path, mtime)`. Invalidate on mtime change. Cache size cap: 50 files.

### Languages supported in v1

- TypeScript / TSX / JavaScript / JSX
- Python
- Go
- Rust

These cover ~85% of the dev-tools-using audience. Other languages fall back to the existing user dictionary unchanged.

### Failure modes and fallback

- **Window title doesn't reveal file path** (some editor configs hide it) → skip biasing, use dictionary only.
- **File path not readable** (sandbox, file moved, file too large >1MB) → skip.
- **Parse error** (tree-sitter throws on malformed file) → cache the failure, don't retry until mtime changes.
- **No tree-sitter grammar for that language** → skip.

All failures are silent. The feature is purely additive — worst case is current behavior.

### Performance

- Parse runs on first dictation against a new file, takes 5–30ms for typical source files. Subsequent dictations are cache-hits, <1ms.
- File-system reads happen off the cleanup hot path: parse triggers on *focus change*, not on hotkey-press. By the time the user dictates, identifiers are already extracted and waiting.
- Hard budget: focus-change parse must complete in 50ms or it gets cancelled. The user's typing fluency comes first.

### Permissions

macOS file-read permission for user-workspace files: most users have already granted this implicitly by giving the editor itself access. We read the same files the editor has open. No new permission prompt for the typical case.

Edge case: if the file is in a sandboxed location Yappr can't access, we silently skip and fall back. Don't prompt the user for Full Disk Access — that's a trust-eroding ask.

### Implementation sketch

- New module `src/main/biasing/editor-context.ts`
- Tree-sitter grammars bundled via `tree-sitter-typescript`, `tree-sitter-python`, etc. (~5MB total, acceptable for a desktop app).
- Window-title regex registry per editor in `src/main/biasing/window-title-parsers.ts`.
- LRU cache `Map<string, {identifiers: string[], mtime: number}>` capped at 50 entries.
- Wire into existing Whisper prompt construction at the point where the user dictionary is currently injected; concat: `[builtin-terms, user-dictionary, editor-identifiers].slice(0, ~200 tokens)`.

### Done means

- Dictating `"call handleSubmit on the form"` into a TSX file containing `handleSubmit` produces `handleSubmit` not `handle submit` or `handles a bit`.
- Cache hit rate >90% measured over a real dev session.
- Zero impact on dictation latency (parse already complete by hotkey-press).

### Out of scope

- LSP-based unsaved-buffer reads (deferred — that's the "better" version in the future-features memo).
- File-tagging output formatting (already done in existing IDE-aware cleanup).
- Languages beyond the v1 set.

## Feature 4 — Context Memory

### What it is

Three stacked context layers passed to the cleanup LLM, giving it a richer picture of who the user is, what they're working on, and what's already in the text box they're dictating into. Goal: better polish quality, fewer "wait that's not what I meant" moments, more consistent voice.

The user's framing: "knows some of the things you talked about... auto-compact... have like maybe a paragraph about them in general."

### The three layers

**Layer A — Focused text-box snapshot.**

At hotkey-release (not press — same timing as app-category detection), read the focused text element's current contents via the macOS Accessibility API: `AXValue` on the focused `AXTextArea` / `AXTextField`. If the read succeeds within 30ms, include up to the last 500 tokens of that text in the cleanup prompt as "user is currently writing this." If the read fails or times out, this layer is silently omitted — the other two layers still work, and the user notices nothing.

No clipboard manipulation. No select-all. No flicker. If AX doesn't expose the text, we don't have the context — and that's fine.

**Layer B — Global user-overview summary.**

A persistent ~paragraph (target: 80–150 words) describing the user in general terms: domains they work in, names they mention often, writing voice, tools they use, recurring topics. Stored as a plain-text string in SQLite at `userData/context.db`.

This is one of the two auto-compacted summaries.

**Layer C — Per-category activity summary.**

A short summary (target: 40–80 words) per app-category (messaging / work / writing / code) covering the user's recent 20 dictations *in that category*. Passed only when dictating into the matching category.

This is the second auto-compacted summary. Stored as 4 rows in SQLite (one per category).

### Compaction strategy

A background Groq call runs after every **20 dictations** total. Logic:

1. Wake on dictation-count-since-last-compaction reaching 20.
2. Delay until system is idle: no dictation in last 60s AND app not currently recording.
3. Fetch the last 20 transcripts globally. Send to llama-3.1-8b-instant with a prompt: "Given the existing user-overview paragraph and these 20 new transcripts, produce a refreshed user-overview paragraph (80–150 words) and a refreshed per-category summary for each of the four categories that appears in these transcripts."
4. Parse, store, reset counter.

The compaction call is the only LLM work tied to context memory and it happens off the hot path. Worst case latency for the user: zero. Worst case cost: one extra 8b-instant call per 20 dictations, billed against the user's own Groq key (BYOK), pennies per month.

### Prompt injection into cleanup

The cleanup prompt assembly gains a new "context" section, included only when populated, structured as:

```
USER CONTEXT (do not echo or summarize):
- Who: {user-overview paragraph, ~120 words}
- Recent in this category: {per-category summary, ~60 words}
- Currently writing: {focused-text-box snippet, ~500 tokens}
```

Existing cleanup prompts (per-strictness, per-category) are otherwise unchanged. The LLM is instructed to use context as background only, never to copy from it. Light-tier (L1, regex-only) does not load context — it doesn't call the LLM at all, so the layers have nothing to inform.

### Performance budget

Hard rules, enforced in code:

- AX read of focused text box: **30ms max**, time-bounded with `Promise.race(read, timeout)`. Time out → omit layer A.
- SQLite read of summaries: **5ms max** under normal load. Use synchronous `better-sqlite3` on the main process; the file is one row per summary, queries are trivial.
- Total context-assembly budget: **40ms**, runs in parallel with audio-encode-and-upload on cleanup-bound transcripts, so net user-visible latency is 0ms in practice.

If any layer breaches its budget, log a counter (visible in the Transparency panel if/when that ships) and drop that layer for this dictation only. Never let context memory slow the hot path.

### Privacy

- Both summaries are **plain text strings on disk**, in the user's profile directory. Never transmitted except as part of the user's own Groq cleanup calls (BYOK).
- One-click "Clear context memory" button in Settings → AI. Wipes both summaries and the dictation count. Idempotent.
- "Pause context memory" toggle in Settings → AI. When off, no compaction runs, no layers are injected. User dictionary and per-app cleanup behavior continue working normally.
- AX text-box reads are never persisted — they live in memory for the duration of one cleanup call.

### Implementation sketch

- New module `src/main/context/store.ts` — wraps better-sqlite3 schema: `user_overview TEXT`, `category_summaries (category TEXT PRIMARY KEY, summary TEXT)`, `dictation_count INTEGER`.
- New module `src/main/context/compactor.ts` — background loop, idle detection, Groq call, summary update.
- New module `src/main/context/ax-reader.ts` — AX `AXValue` read with hard timeout. Native helper via existing accessibility bridge.
- New module `src/main/context/prompt-injector.ts` — builds the context section, enforces the 40ms total budget.
- Wire into cleanup pipeline at the point where strictness-tier and category-prompt are selected.
- Settings → AI tab gains: "Pause context memory" toggle, "Clear context memory" button, last-compaction timestamp display.

### Done means

- Dictating into Slack about "the Q3 roadmap" three days after the user has been working on Q3 roadmap docs produces cleanup that knows "Q3 roadmap" is a thing without the user typing it.
- Compaction runs every 20 dictations, verifiable in logs.
- Measured hot-path latency unchanged from current main branch (±5ms tolerance).
- Toggling "Pause context memory" off → no LLM call, no summary used. Verifiable in network logs.

### Out of scope

- Cross-machine sync of summaries (will be a monthly-tier feature).
- Surfacing summaries to the user as readable text (future — could be a Settings panel viewer).
- LLM-judged "is this dictation worth remembering" filtering (every dictation counts equally).
- Vector-search over historical dictations (overkill for the value).

### Risks

- **LLM mis-compaction:** summaries drift into nonsense after enough rewrites. Mitigation: the compaction prompt anchors on the *previous* summary as the spine, treating the new 20 transcripts as additive. Add a periodic "from scratch" rebuild every N compactions (suggest N=10, so every 200 dictations).
- **Stale context after job change:** user switches domains, summary stays old for ~20 dictations. Mitigation: the compaction prompt instructs the LLM to *replace* topics that no longer appear, not just append. Manual "Clear context memory" is the user escape hatch.
- **AX permission edge cases:** some Electron apps (Discord, Slack) return empty `AXValue` for chat composer. Acceptable — feature degrades to two layers in those apps.

## Suggested launch order

1. **Lifetime tier** — ships first. Sets up the pricing surface for everything else. Mostly Stripe + landing-page work, low engineering risk.
2. **Pre-roll buffer + BT robustness** — ships second. Pure win, no UX changes, fixes Wispr-shared failure modes. Builds reliability narrative.
3. **Editor-aware biasing (quick-win)** — ships third. Largest accuracy delta for the developer audience that converts best to paid. Adds tree-sitter as a dependency, used again in feature 4 if we ever do LSP integration.
4. **Context memory** — ships fourth. Largest scope, most performance risk, most privacy surface. Wants the other three already in production so the launch story is "we ship reliably."

Nothing here forces this order — each section is implementation-independent.

## Non-goals for this entire pack

- Mobile (iOS/Android).
- Windows-specific features (Windows packaging exists, but these four are Mac-first).
- Multi-user / team / org features.
- Telemetry of any kind. No layer adds opt-in or opt-out network calls beyond the user's own Groq inference path.
